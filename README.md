# scrapper-trends

> Autonomous, multi-niche content pipeline that turns Google-style trending
> topics into SEO-ready, fact-checked, image-illustrated WordPress posts —
> generated by an LLM, reviewed by an LLM, and published on a schedule with
> zero human intervention.

`scrapper-trends` is a production system that runs three independent news blogs
(entertainment, sports, finance) end-to-end: it discovers what's trending,
reads the underlying source articles, writes an original article grounded in
verified facts, attaches a relevant cover image, runs a second AI pass to catch
glaring errors, and publishes to WordPress — all driven by GitHub Actions cron
with no servers to maintain.

## What it does

For each niche, on every scheduled run the system:

1. **Discovers trends** — scrapes a trending-topics page with a headless browser
   (Playwright/Chromium) and walks the trend table row by row.
2. **Follows the sources** — for each trend it extracts up to 3 source article
   URLs and scrapes a cover image from the article page.
3. **Writes an original article** — an LLM (Anthropic Claude by default, Gemini
   optionally) generates a complete, SEO-optimized post in HTML, **grounded with
   live web search** so facts (names, dates, relationships) are verified before
   anything is written.
4. **Reviews itself** — a second, lighter LLM pass uses **vision** to confirm the
   cover image matches the article and does a conservative sanity check for
   glaring factual errors.
5. **Publishes to WordPress** — uploads the image, creates the post via the WP
   REST API (live, or as a *draft* if the review flagged it), sets SEO metadata
   (Yoast), assigns a category and a random author, and pings a Telegram channel.
6. **Remembers what it published** — records source URLs so the same trend is
   never regenerated, keeping AI spend down.

Each run publishes **at most one post and exits** — scheduling cadence is owned
entirely by cron, which keeps the runtime simple, idempotent, and cheap.

## Highlights for reviewers

If you're evaluating this as a work sample, here's what it demonstrates:

- **LLM application engineering, end to end** — prompt design, tool use
  (server-side `web_search`), multimodal/vision review, JSON extraction from free
  text, retry/backoff against rate limits, and a **provider abstraction** that
  swaps Anthropic ↔ Gemini behind one config object.
- **AI safety/quality patterns** — generation is *grounded* (no hallucinated
  facts), and an **LLM-as-judge** acts as a *soft gate*: flagged posts become
  drafts for a human instead of being silently dropped or blindly published.
- **Cost-conscious design** — a local state file dedupes trends *before* any AI
  call; web search is bounded; WordPress slugs (not the state file) are the
  correctness source of truth, so a lost cache can only ever cause a wasted
  generation, never a duplicate live post.
- **Serverless ops** — the whole thing runs on **GitHub Actions cron** with a
  reusable workflow, dependency/browser caching, and a concurrency-safe state
  commit that **merges at the data level** to survive concurrent runs racing on
  the same branch.
- **Clean architecture** — one shared engine (`scrapper_base.py`) + thin
  per-niche config scripts (DRY), pure helper functions, custom exception types,
  rotating file logs, and **34 unit tests**.
- **Web scraping in practice** — robust image extraction (`srcset` → `src`
  fallback, resolution upgrades), domain blocklists, and graceful row-by-row
  fallback when a source page misbehaves.

## How it works

`run_once(config)` performs exactly one publish cycle:

1. **Commercial-hours gate** — pauses publishing between 01:00–05:00 in the
   niche's timezone (`America/Sao_Paulo` by default). Cron fires in UTC, so this
   gate, not the cron, owns the quiet window; off-hours runs just exit 0.
2. **Recover WordPress data** — fetches recent posts/categories (for internal
   links) and the set of existing slugs (the dedup source of truth).
3. **Find a publishable trend** — opens Playwright once and walks trend rows:
   - Extract up to 3 source URLs per trend.
   - If any URL is already in `state/<niche>.json`, **skip with no AI call**.
   - Otherwise scrape the cover image and generate the article (grounded with
     web search). If the resulting slug already exists in WordPress, record it
     and move on.
4. **Soft content review** — a light, vision-enabled pass on the same provider
   flags an unrelated cover image or a *glaring* factual error. It **fails open**
   (publishes as usual if review is disabled or errors). A flagged post is still
   created — as a **draft** for human review.
5. **Publish** — upload image → create post → clean up → notify Telegram, with
   up to 5 retries.
6. **Persist state** — append the published source URLs (capped history). In CI
   this file is committed back to the repo.

## Architecture

Each entry script (`pw_trends*.py`) is a thin config that builds a `NicheConfig`
and calls `run_once(config)`. All shared logic lives in `scrapper_base.py`.

```
pw_trends.py ─────────┐
pw_trends_sports.py ──┼──► NicheConfig ──► run_once() ──► scrapper_base.py
pw_trends_finance.py ─┘                                        │
                                                               ├─ Playwright scrape (trends + cover image)
                                                               ├─ LLM generate (grounded w/ web search)
                                                               ├─ LLM review   (vision soft gate)
                                                               ├─ WordPress REST (publish / draft)
                                                               ├─ Telegram notify
                                                               └─ state/<niche>.json (dedup cache)
```

`NicheConfig` (a frozen dataclass) captures everything a niche differs on:
WordPress credentials, Telegram token, trends URL, the niche label used in the
prompt, the category-selection function, AI provider/models, timezone, and
review toggle.

|---|---|---|---|---|
| `pw_trends.py` | Entertainment (*Fofocando*) | `TELEGRAM_TOKEN` | every 2h | 9 (Novelas) if the title matches the novela regex, else 6 (Notícias) |

| `pw_trends_finance.py` | Finance | `TELEGRAM_FINANCE_TOKEN` | every 4h | Category 1 |

The AI provider and models are shared across niches via env vars; any niche can
override by passing the argument explicitly to `NicheConfig`.

## Getting started

```bash
# 1. Install dependencies
pip install -r requirements.txt
python -m playwright install chromium

# 2. Create a .env file (see Configuration below)

# 3. Publish ONE post for a niche, then exit
python pw_trends.py          # Entertainment/gossip (blog: Fofocando)
python pw_trends_finance.py  # Finance
python pw_trends_sports.py   # Sports
```

Each script is single-shot: it publishes at most one post and exits 0 (or
non-zero on a hard publish failure). Scheduling is owned by the GitHub Actions
workflows under `.github/workflows/`.

## Configuration

The same variable names are read from `.env` locally and from **GitHub Secrets**
in CI.

```ini
# --- AI ---
ANTHROPIC_API_KEY=         # required when AI_PROVIDER=anthropic (default)
GEMINI_API_KEY=            # required when AI_PROVIDER=gemini
AI_PROVIDER=               # "anthropic" (default) or "gemini"
AI_MODEL=                  # generation model (default: claude-sonnet-4-6)
REVIEW_MODEL=              # review model    (default: claude-sonnet-4-6)

# --- WordPress (one set per blog) ---
WP_BLOG_FOFOCANDO_URL=
WP_BLOG_FOFOCANDO_USER=
WP_BLOG_FOFOCANDO_PASS=
WP_BLOG_SPORT_URL=
WP_BLOG_SPORT_USER=
WP_BLOG_SPORT_PASS=
WP_BLOG_FINANCE_URL=
WP_BLOG_FINANCE_USER=
WP_BLOG_FINANCE_PASS=

# --- Telegram ---
TELEGRAM_TOKEN=
TELEGRAM_SPORT_TOKEN=
TELEGRAM_FINANCE_TOKEN=
TELEGRAM_CHAT_ID=

# --- Trend sources ---
TRENDS_URL_ENTERTEINMENT=
TRENDS_URL_SPORTS=
TRENDS_URL_FINANCE=
```

For CI, add every value above as a repository **Secret**.

## Scheduling & CI/CD

One workflow per niche (`entertainment.yml`, `sports.yml`, `finance.yml`) calls
the reusable `_publish.yml`, which:

- checks out the repo and sets up Python 3.13 with pip caching,
- caches the Playwright Chromium browser,
- installs deps and runs the niche script,
- commits the updated `state/` file back to the repo.

Crons are defined in **UTC**; the script's commercial-hours gate handles the
01:00–05:00 pause, so cron can fire 24/7 (off-hours runs exit 0). A
`concurrency` group prevents two runs of the same niche from overlapping, and
the state commit syncs to the remote tip and **unions hrefs at the data level**
(`.github/scripts/merge_state.py`) so concurrent niche runs pushing to the same
branch don't clobber each other.

## Design decisions worth noting

- **Single-shot scripts, cron-driven cadence.** No long-running loop means no
  process to babysit, trivial idempotency, and per-run isolation in CI.
- **WordPress slugs are the correctness source of truth.** The local state file
  is purely a cost optimization; losing it can at worst cause a duplicate
  *generation*, never a duplicate *published post*.
- **Grounded generation + soft review.** Facts are verified at write time via web
  search; the review is intentionally light (vision for cover-image relevance +
  glaring-error sanity check) and **fails open** so a reviewer outage never
  blocks the pipeline.
- **Soft blocking over hard blocking.** A flagged post is published as a draft
  for a human, not discarded — favoring throughput while keeping a safety net.
- **Provider-agnostic AI layer.** Anthropic and Gemini are interchangeable
  behind `NicheConfig`, including their different grounding and vision APIs.
